# План улучшений Fast Promo Videos Bot

## Цель

Сделать бота устойчивее в продакшене, проще в поддержке и сильнее как продукт: меньше ручного разбора инцидентов, меньше потерянных сценариев после сбоев, больше прозрачности для пользователя и владельца.

## Приоритет 1: операционная надежность

### 1. Recovery зависших задач

Добавить процесс восстановления задач, которые остались в статусе `processing` слишком долго.

Что сделать:

- хранить `started_at` и `updated_at` у задач монтажа;
- считать задачу зависшей, если она в `processing` дольше заданного лимита;
- добавить периодический recovery job;
- для зависших задач либо возвращать видео на баланс, либо повторно ставить задачу в очередь;
- логировать причину восстановления.

Зачем:

- пользователь не теряет видео из-за падения worker;
- владелец видит, какие задачи ломаются;
- после рестартов система сама приводит себя в нормальное состояние.

### 2. Перенести pending-состояние из памяти в PostgreSQL или Redis

Сейчас незавершенный сценарий до отправки в очередь хранится в `pending_videos`.

Что сделать:

- создать таблицу или Redis-структуру для черновиков монтажа;
- сохранять выбранное видео, баннер, текст и настройки;
- добавлять TTL для незавершенных сценариев;
- после рестарта бота продолжать сценарий или вежливо просить пользователя начать заново с понятным объяснением.

Зачем:

- рестарт бота не ломает пользовательский путь;
- проще разбирать обращения в поддержку;
- состояние становится наблюдаемым.

### 3. Идемпотентность worker-задач

Сделать обработку задач безопасной при повторном запуске.

Что сделать:

- проверять статус задачи перед списанием и монтажом;
- явно разделить состояния `queued`, `processing`, `done`, `failed`, `refunded`;
- не списывать баланс повторно при ретраях;
- хранить внешний `job_id` и внутренний `operation_id`;
- сделать повторную отправку результата безопасной.

Зачем:

- ретраи не создают двойных списаний;
- проще чинить отдельные задачи вручную;
- меньше пограничных багов при падениях.

### 4. Healthcheck для всех сервисов

Сейчас web имеет `GET /health`, но полезно проверять всю цепочку.

Что сделать:

- добавить проверку подключения к PostgreSQL;
- добавить проверку Redis;
- добавить отдельный worker heartbeat;
- показывать версию приложения и время старта;
- использовать healthcheck в `docker-compose`.

Зачем:

- быстрее понятно, что именно упало;
- можно подключить внешний мониторинг;
- деплой становится спокойнее.

## Приоритет 2: тесты

### 1. Smoke-тест Robokassa Receipt

Проверять, что платежная форма Robokassa собирается корректно.

Что покрыть:

- позиция номенклатуры присутствует в `Receipt`;
- `Receipt` участвует в подписи;
- `Receipt` URL-кодируется;
- `sno` не передается;
- сумма, invoice id и описание совпадают с тарифом.

### 2. Unit-тесты тарифов

Проверить все пакеты видео и цены.

Что покрыть:

- 10 видео за 99 рублей;
- 25 видео за 229 рублей;
- 50 видео за 399 рублей;
- 100 видео за 699 рублей;
- корректность скидок в отображении;
- отсутствие покупки несуществующего тарифа.

### 3. Unit-тесты подписи Robokassa

Что покрыть:

- подпись платежной формы;
- подпись `ResultURL`;
- отказ при неверной подписи;
- идемпотентность повторного `ResultURL`;
- отказ при несовпадении суммы или invoice id.

### 4. Тесты баланса

Что покрыть:

- стартовый бонус начисляется один раз;
- покупка увеличивает баланс один раз;
- монтаж списывает ровно одно видео;
- техническая ошибка worker возвращает видео;
- пользователь не может уйти в отрицательный баланс.

### 5. Интеграционный smoke-тест очереди

Минимальный тест цепочки:

- создать пользователя;
- выдать баланс;
- создать задачу;
- поставить в очередь;
- обработать тестовым worker;
- перевести задачу в `done`;
- проверить итоговый баланс и статус.

## Приоритет 3: платежи и поддержка

### 1. Команда поддержки для платежей

Добавить админскую команду для просмотра платежа.

Возможные команды:

- `/admin_payment <invoice_id>`;
- `/admin_user <telegram_id>`;
- `/admin_balance <telegram_id>`;
- `/admin_refund <job_id>`.

Что показывать:

- пользователь;
- сумма;
- пакет;
- статус;
- дата создания;
- дата оплаты;
- Robokassa operation id, если есть;
- был ли баланс начислен.

### 2. Показывать пользователю ID операции

При ошибках и после оплаты показывать короткий ID.

Пример:

```text
ID операции: 12345
```

Зачем:

- пользователь может отправить ID в поддержку;
- владельцу проще найти платеж или задачу;
- меньше переписок в стиле "я вчера оплатил".

### 3. Страница статуса платежа

Добавить простой endpoint или команду для проверки платежа.

Что сделать:

- `/payment_status <invoice_id>` в боте;
- или страница `/robokassa/status/{invoice_id}`;
- показывать `pending`, `paid`, `failed`, `expired`.

### 4. Обработка просроченных pending-платежей

Что сделать:

- добавить `expires_at`;
- периодически переводить старые `pending` в `expired`;
- не начислять баланс по просроченному платежу без дополнительной проверки.

## Приоритет 4: UX в Telegram

### 1. Понятный экран баланса

Добавить команду `/balance`.

Показывать:

- текущий баланс;
- сколько видео можно создать;
- кнопку покупки пакета;
- краткую историю последних операций.

### 2. История операций

Добавить `/history`.

Показывать последние события:

- стартовый бонус;
- покупка;
- списание за монтаж;
- refund;
- техническая ошибка;
- готовое видео.

### 3. Улучшить ошибки пользователя

Сделать отдельные тексты для частых случаев:

- видео слишком большое;
- видео слишком длинное;
- баннер слишком большой;
- текст длиннее 60 символов;
- недостаточно баланса;
- монтаж временно недоступен;
- оплата еще не подтверждена.

### 4. Возврат к незавершенному сценарию

Если пользователь остановился на середине:

- напомнить, что уже загружено;
- дать кнопки `Продолжить`, `Начать заново`, `Отменить`;
- очищать старые черновики по TTL.

### 5. Превью перед списанием

Перед постановкой в очередь показывать финальное подтверждение:

- видео принято;
- баннер принят;
- рекламный текст;
- стоимость: 1 видео;
- текущий баланс после списания.

## Приоритет 5: наблюдаемость

### 1. Структурные логи

Перейти к логам с понятными полями:

- `user_id`;
- `invoice_id`;
- `job_id`;
- `status`;
- `duration_ms`;
- `error_type`.

### 2. Метрики

Собирать базовые показатели:

- сколько пользователей нажали `/start`;
- сколько создали первое видео;
- сколько оплатили;
- конверсия в оплату;
- среднее время монтажа;
- процент ошибок worker;
- сумма оплат за день;
- количество refund.

### 3. Уведомления владельцу

Добавить админские уведомления в Telegram:

- новая оплата;
- ошибка worker;
- зависшая задача восстановлена;
- превышен лимит ошибок за час;
- Robokassa прислала некорректную подпись.

## Приоритет 6: хранение и восстановление результата

### 1. Object storage для готовых видео

Сейчас видео после отправки не восстановить.

Что можно сделать:

- подключить S3-совместимое хранилище;
- хранить готовые видео ограниченное время, например 24 или 72 часа;
- давать пользователю кнопку `Скачать повторно`;
- автоматически удалять старые файлы.

Зачем:

- меньше потерь, если Telegram не доставил файл;
- проще поддерживать платящих пользователей;
- можно добавить историю готовых видео.

### 2. Настраиваемый TTL файлов

Для временных файлов:

- отдельно TTL исходников;
- отдельно TTL результатов;
- отдельная очистка зависших папок `tmp/<job_id>`.

## Приоритет 7: безопасность и антиабьюз

### 1. Rate limits

Ограничить частые действия:

- загрузка видео;
- создание платежей;
- повторные команды;
- ошибки ввода.

### 2. Лимиты на пользователя

Добавить дневные лимиты для бесплатного использования:

- число попыток загрузки;
- число неудачных обработок;
- число неоплаченных pending-платежей.

### 3. Проверка типов файлов

Проверять не только расширение, но и фактический MIME/codec через `ffprobe`.

### 4. Защита Robokassa endpoints

Проверить:

- строгую валидацию подписи;
- отказ при неизвестном `invoice_id`;
- отказ при несовпадении суммы;
- логирование подозрительных запросов;
- отсутствие лишних данных в ответах.

## Приоритет 8: продукт и монетизация

### 1. Онбординг

Сделать первое использование максимально коротким:

- `/start`;
- показать баланс и бонус;
- сразу предложить загрузить видео;
- после первого результата предложить купить пакет.

### 2. Демо-режим

Можно добавить пример результата:

- короткое демо-видео;
- описание формата;
- кнопка `Создать свое`.

### 3. Промокоды

Добавить простую систему промокодов:

- фиксированное число видео;
- срок действия;
- лимит активаций;
- активация один раз на пользователя.

### 4. Реферальный бонус

Если будет органический спрос:

- пользователь получает ссылку;
- приглашенный получает бонус;
- пригласивший получает бонус после первого созданного видео или первой оплаты.

### 5. Пакеты для частых пользователей

Добавить крупный пакет или подписку:

- 250 видео;
- 500 видео;
- месячный лимит;
- ручная выдача корпоративным клиентам.

## Приоритет 9: администрирование

### 1. Мини-админка

Не обязательно делать полноценный web-интерфейс сразу. Можно начать с Telegram-команд.

Минимум:

- список последних оплат;
- список последних ошибок;
- поиск пользователя;
- ручное начисление;
- ручной refund;
- просмотр зависших задач.

### 2. Экспорт данных

Добавить выгрузки:

- оплаты за период;
- пользователи;
- операции баланса;
- ошибки worker.

Формат: CSV.

### 3. Роли админов

Если админов станет больше одного:

- owner;
- support;
- readonly.

## Приоритет 10: качество монтажа

### 1. Профили рендера

Добавить несколько вариантов:

- быстрый;
- стандартный;
- лучшее качество.

### 2. Улучшенная обработка ошибок ffmpeg

Что сделать:

- классифицировать ошибки;
- писать понятную причину;
- отличать плохой файл пользователя от технической ошибки сервера;
- при плохом файле не всегда делать refund, если обработка даже не стартовала.

### 3. Проверка результата перед отправкой

После рендера проверять:

- файл существует;
- размер больше минимального порога;
- длительность не нулевая;
- разрешение 1080x1920;
- есть аудиодорожка, если она ожидается.

## Рекомендуемый порядок работ

1. Тесты Robokassa: `Receipt`, подпись, идемпотентность.
2. Тесты баланса: бонус, покупка, списание, refund.
3. Recovery зависших `processing` задач.
4. Перенос `pending_videos` из памяти в PostgreSQL или Redis.
5. Админские команды для платежей, пользователей и задач.
6. Команды `/balance` и `/history`.
7. Структурные логи и уведомления владельцу.
8. Object storage для временного восстановления готовых видео.
9. Rate limits и антиабьюз.
10. Промокоды, рефералка и новые пакеты.

## Самый короткий полезный спринт

Если делать только один небольшой, но сильный апгрейд, я бы взял такой набор:

1. Smoke-тест Robokassa `Receipt`.
2. Unit-тесты подписи Robokassa.
3. Unit-тесты операций баланса.
4. Recovery зависших задач.
5. Команду `/admin_payment <invoice_id>`.

Это быстро повысит уверенность в деньгах, оплатах и восстановлении после сбоев.
