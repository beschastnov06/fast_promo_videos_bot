# Схема оплаты и чеков

## Цель

Добавить оплату через Robokassa в Fast Promo Videos Bot и автоматически начислять видео на баланс пользователя после успешной оплаты.

Оплаты принимаются через Robokassa. Чеки для самозанятого формируются через сервис Robokassa `Робочеки СМЗ`.

## Что продаем

Бот продает пакеты видео, а не абстрактные кредиты.

| Пакет | Цена | Начисление | Номенклатура в чеке |
| --- | ---: | ---: | --- |
| 10 видео | 99 ₽ | +10 видео | Пакет 10 видео Fast Promo Videos Bot |
| 25 видео | 229 ₽ | +25 видео | Пакет 25 видео Fast Promo Videos Bot |
| 50 видео | 399 ₽ | +50 видео | Пакет 50 видео Fast Promo Videos Bot |
| 100 видео | 699 ₽ | +100 видео | Пакет 100 видео Fast Promo Videos Bot |

Одно видео в рамках пакета дает право обработать одно видео длительностью до 60 секунд. Видео длиннее 60 секунд бот отклоняет и не отправляет в монтаж.

## Пользовательский сценарий

1. Пользователь открывает `/buy` или главное меню.
2. Бот показывает текущий баланс и список пакетов.
3. Перед оплатой бот показывает текст:

   ```text
   Производя оплату, вы соглашаетесь с офертой и условиями обработки персональных данных:
   https://telegra.ph/Oferta-usloviya-okazaniya-uslug-i-politika-obrabotki-personalnyh-dannyh-05-07
   ```

4. Пользователь выбирает пакет.
5. Бот создает локальную запись платежа со статусом `pending`.
6. Бот отправляет ссылку на оплату Robokassa.
7. Пользователь оплачивает на странице Robokassa.
8. Robokassa вызывает наш `ResultURL`.
9. Backend проверяет подпись Robokassa.
10. Backend переводит платеж в статус `paid` и начисляет видео на баланс пользователя.
11. Бот отправляет подтверждение:

    ```text
    Оплата прошла ✅
    Начислено: 25 видео
    Баланс: 28 видео

    Чек будет сформирован сервисом Робочеки СМЗ и отправлен покупателю.
    ```

12. Пользователю отправляется чек. Базовый сценарий: покупатель вводит email на стороне Robokassa, а Robokassa/`Робочеки СМЗ` отправляют чек на этот email. Если Robokassa покажет чек или ссылку на чек сразу после оплаты на своей странице, используем это как лучший UX. Если Robokassa даст нам ссылку на чек через API или callback, бот дополнительно отправляет эту ссылку в Telegram.

## Создание платежа Robokassa

Ссылки на оплату создает наш backend. Не используем свободные ссылки на произвольную сумму из кабинета Robokassa, потому что для корректного формирования чеков Robokassa требует передавать состав корзины, то есть номенклатуру.

Основные параметры Robokassa:

| Параметр | Назначение |
| --- | --- |
| `MerchantLogin` | Логин магазина Robokassa |
| `OutSum` | Сумма платежа, например `229.00` |
| `InvId` | Внутренний числовой ID счета/платежа |
| `Description` | Описание платежа |
| `SignatureValue` | Контрольная подпись запроса |
| `Receipt` | JSON с фискальными данными и номенклатурой |
| `Email` или телефон | Контакт покупателя для доставки чека. Основной вариант: пользователь вводит email на стороне Robokassa |
| `IsTest` | `1` для тестового режима, `0` для боевого |
| пользовательские поля | Связка платежа с пользователем/заказом, если понадобится |

Если передаем `Receipt`, он участвует в подписи:

```text
MerchantLogin:OutSum:InvId:Receipt:Пароль#1
```

Из-за размера `Receipt` Robokassa рекомендует использовать `POST`, а не длинный `GET`.

Источники:

- Интерфейс оплаты Robokassa: https://docs.robokassa.ru/ru/pay-interface
- Фискализация Robokassa: https://docs.robokassa.ru/ru/fiscalization
- Уведомления Robokassa: https://docs.robokassa.ru/ru/notifications-and-redirects

## Номенклатура для чека

На каждый платеж передаем одну позицию: купленный пакет видео.

Пример для пакета на 25 видео:

```json
{
  "sno": "nps",
  "items": [
    {
      "name": "Пакет 25 видео Fast Promo Videos Bot",
      "quantity": 1,
      "sum": 229,
      "payment_method": "full_payment",
      "payment_object": "service",
      "tax": "none"
    }
  ]
}
```

Примечания:

- `payment_object` должен быть `service`, потому что мы продаем услугу обработки видео.
- `quantity` ставим `1`, потому что пакет продается как одна услуга.
- `sum` должна совпадать с `OutSum`.
- `tax` и `sno` надо дополнительно проверить в личном кабинете/документации Robokassa именно для `Робочеки СМЗ`. Для самозанятых на НПД часть полей 54-ФЗ может маппиться Robokassa иначе, чем для обычной онлайн-кассы.

## Обработка ResultURL

`ResultURL` — серверное уведомление Robokassa об успешной оплате.

Backend должен:

1. Прочитать `OutSum`, `InvId`, `SignatureValue` и пользовательские поля.
2. Найти платеж по `InvId`.
3. Проверить, что платеж еще в статусе `pending`.
4. Пересчитать подпись ResultURL через `Пароль#2`.
5. Сравнить подписи безопасным способом.
6. Проверить, что `OutSum` совпадает с ожидаемой ценой пакета.
7. Перевести платеж в статус `paid`.
8. Начислить пакет видео на баланс пользователя.
9. Сохранить `paid_at`, ID операции Robokassa, если он пришел, и сырые данные callback для диагностики.
10. Вернуть Robokassa простой текст:

    ```text
    OK{InvId}
    ```

Robokassa требует, чтобы обработчик `ResultURL` после успешной проверки вернул `OK` плюс номер счета.

## SuccessURL и FailURL

`SuccessURL` и `FailURL` — это редиректы пользователя после оплаты. Их нельзя использовать как источник истины для начисления видео.

Баланс обновляем только через `ResultURL` после проверки подписи.

Рекомендованное поведение:

- `SuccessURL`: показать страницу "Оплата подтверждается, вернитесь в Telegram". Если Robokassa умеет показать чек или ссылку на чек сразу после оплаты, предпочитаем этот сценарий на стороне Robokassa.
- `FailURL`: показать страницу "Оплата отменена или не прошла, вернитесь в Telegram".

Для Telegram-бота это могут быть минимальные backend-страницы или простые статические HTML-страницы.

## Изменения в базе данных

Текущая модель `payments` подходит как основа, но перед production ее лучше расширить.

Рекомендуемые поля:

| Поле | Назначение |
| --- | --- |
| `id` | Внутренний UUID |
| `invoice_id` | Числовой `InvId` для Robokassa |
| `user_id` | Пользователь, которому начисляем видео |
| `provider` | `robokassa` |
| `provider_payment_id` | ID операции Robokassa / `OpKey`, если доступен |
| `status` | `pending`, `paid`, `failed`, `cancelled`, `refunded` |
| `amount_cents` | Ожидаемая сумма в копейках |
| `currency` | `RUB` |
| `credits_amount` | Количество видео к начислению |
| `package_code` | `videos_10`, `videos_25` и т.д. |
| `receipt_status` | `unknown`, `pending`, `sent`, `failed` |
| `receipt_url` | Ссылка на чек, если Robokassa даст ее через API |
| `raw_provider_payload` | Сырые данные callback для диагностики |
| `created_at`, `paid_at` | Временные метки |

Изменение статуса платежа и начисление баланса должны происходить в одной транзакции, чтобы не было двойного начисления.

## Идемпотентность

Robokassa может повторно вызывать `ResultURL`, поэтому обработчик должен быть идемпотентным.

Если платеж уже `paid`, возвращаем `OK{InvId}` и не начисляем видео повторно.

Если подпись или сумма не прошли проверку, баланс не пополняем.

## Чеки и "Мой налог"

Не строим первую версию вокруг сценария "бот забирает чек из Мой налог".

Почему:

- У "Мой налог" нет простого публичного API, через который наш бот мог бы стабильно получать ссылки на чеки после каждой оплаты.
- Robokassa пишет, что `Робочеки СМЗ` подключаются к кабинету продавца в "Мой налог" и автоматически создают чеки на каждую оплату через Robokassa.
- Robokassa также пишет, что чек отправляется покупателю и в налоговую.

Источник:

- Робочеки СМЗ: https://robokassa.com/online-check/robocheck-smz/

MVP-сценарий по чекам:

1. Включить `Робочеки СМЗ` в Robokassa.
2. Одобрить подключение Robokassa в "Мой налог".
3. Передавать корректную номенклатуру в `Receipt` для каждого платежа.
4. Дать Robokassa/Робочекам сформировать и отправить чек.
5. Убедиться, что пользователь получает чек через Robokassa/`Робочеки СМЗ` на email/телефон, указанный при оплате.
6. Бот после оплаты сообщает, что чек сформирован/будет сформирован сервисом `Робочеки СМЗ` и отправлен покупателю.

Если Robokassa даст в нашем аккаунте/API ссылку на чек или статус фискализации, добавим второй этап:

1. Хранить статус/ссылку на чек в `payments`.
2. Получать статус через API или callback, если такой сценарий доступен.
3. Отправлять ссылку на чек пользователю в Telegram.

Пока это не подтверждено в нашем аккаунте Robokassa/API, не обещаем пользователю, что бот всегда пришлет прямую ссылку на чек из "Мой налог" в Telegram.

## Контакт покупателя для доставки чека

`Робочеки СМЗ` должны понимать, куда отправлять чек покупателю.

Основной сценарий для MVP:

1. Бот не спрашивает email до оплаты.
2. Пользователь вводит email на платежной странице Robokassa.
3. Robokassa использует этот email для отправки чека и уведомлений.
4. В `ResultURL` Robokassa может вернуть `EMail`, указанный покупателем при оплате.

Так мы меньше собираем персональных данных в боте и перекладываем ввод контактных данных на платежную форму Robokassa.

Запасной сценарий:

Если окажется, что Robokassa не показывает обязательное поле email на платежной странице при `Робочеки СМЗ`, тогда либо включаем это поле в настройках/через поддержку Robokassa, либо временно спрашиваем email в боте перед созданием платежа и передаем его в параметре `Email`.

## Ошибочные сценарии

| Сценарий | Поведение |
| --- | --- |
| Пользователь открыл ссылку, но не оплатил | Платеж остается `pending`, баланс не меняется |
| Robokassa вызвала `FailURL` | Показываем "оплата отменена/не прошла", баланс не меняется |
| Неверная подпись `ResultURL` | Отклоняем callback, баланс не меняется |
| Сумма в `ResultURL` не совпала | Логируем подозрительный платеж, баланс не меняется |
| Повторный `ResultURL` по уже оплаченному счету | Возвращаем `OK{InvId}`, видео повторно не начисляем |
| Чек не сформировался | Платеж остается оплаченным, логируем и решаем через Robokassa/support, потому что фискализация делегирована `Робочеки СМЗ` |


Секреты нельзя коммитить в репозиторий.

## Чеклист реализации

1. Добавить переменные конфигурации Robokassa.
2. Добавить/расширить поля модели платежей.
3. Завести каталог пакетов в коде на основе текущих тарифов.
4. Добавить генерацию ссылки Robokassa с `Receipt`.
5. Добавить endpoint для `ResultURL`.
6. Проверять подпись и сумму в `ResultURL`.
7. Сделать идемпотентное начисление баланса.
8. Добавить сообщения бота для создания платежа и подтверждения оплаты.
9. Добавить текст согласия с офертой перед оплатой.
10. Включить `Робочеки СМЗ` в Robokassa и подтвердить подключение в "Мой налог".
11. Провести одну минимальную боевую оплату и проверить:
    - платеж успешно проходит;
    - баланс начисляется ровно один раз;
    - чек появляется в "Мой налог";
    - покупатель получает чек через Robokassa/Робочеки;
    - бот отправляет подтверждение оплаты.
